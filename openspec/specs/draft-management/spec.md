## ADDED Requirements

### Requirement: Draft created from workflow approval
The system SHALL automatically create a Draft document when a user approves a writing workflow; the draft is linked to its WorkflowRun, the source article (if any), and stores the selected title and title candidates.

#### Scenario: Draft auto-created on workflow approval
- **WHEN** user approves a writing WorkflowRun and selects a title
- **THEN** system creates a Draft with the final content, selected title, title_candidates array (all 3 from Editor), status="draft", workflow_run_id set, source_article_ids set (list of all reference article IDs from the WorkflowRun input), and navigates frontend to /drafts/{id}

### Requirement: Draft stores title candidates
The Draft document SHALL store the 3 title candidates generated by the Editor Agent, allowing the user to switch between them in the editor even after approval.

#### Scenario: Title candidates accessible in editor
- **WHEN** user opens /drafts/{id}
- **THEN** editor shows the current title with a "title candidates" panel listing all 3 options; clicking a candidate updates Draft.title

### Requirement: Markdown editor
The system SHALL provide a Markdown editor (not rich text) for editing draft content; the editor SHALL support standard Markdown syntax with a live preview pane.

#### Scenario: Draft edited in Markdown
- **WHEN** user opens /drafts/{id}
- **THEN** page displays a split-pane Markdown editor (edit on left, rendered preview on right) with syntax highlighting

#### Scenario: Auto-save on change
- **WHEN** user stops typing for 1 second
- **THEN** system auto-saves the draft content via PATCH /drafts/{id} and shows a "saved" indicator

### Requirement: Paragraph-level rewrite (去AI味)
The system SHALL allow users to select any text span in the draft editor and trigger an on-demand rewrite by the Editor Agent to reduce AI-like patterns in that specific section.

This requires a dedicated backend endpoint: POST /drafts/{id}/rewrite-section

#### Scenario: User rewrites a selected paragraph
- **WHEN** user selects text in the editor and clicks "去AI味" button
- **THEN** frontend calls POST /drafts/{id}/rewrite-section with the selected text and surrounding context
- **THEN** backend invokes Editor Agent on the selection, returns rewritten text
- **THEN** frontend displays a diff view (original vs rewritten) and user can accept or discard

#### Scenario: Rewrite replaces selected text on accept
- **WHEN** user accepts the rewrite suggestion
- **THEN** the selected text in the editor is replaced with the rewritten version and auto-save triggers

### Requirement: Version history
The system SHALL append an entry to edit_history before each content save; users can browse and restore any previous version.

#### Scenario: Edit saves version history
- **WHEN** user saves an edit (manually or auto-save)
- **THEN** system appends the previous content to edit_history before overwriting

#### Scenario: Restore previous version
- **WHEN** user clicks a version in the history sidebar and chooses "Restore"
- **THEN** editor content is replaced with that version's content and a new history entry is created marking the restore

### Requirement: List drafts
The system SHALL provide a paginated list of drafts with filtering by status and sorting by created_at.

#### Scenario: List pending drafts
- **WHEN** user requests GET /drafts?status=draft
- **THEN** system returns all drafts with status="draft" sorted by created_at descending

### Requirement: Approve draft
The system SHALL allow marking a draft as approved from the editor page.

#### Scenario: Approve draft from editor
- **WHEN** user clicks "标记为已发布" button on /drafts/{id}
- **THEN** POST /drafts/{id}/approve sets status to "approved" and shows confirmation

### Requirement: Delete draft
The system SHALL allow permanently deleting a draft; associated WorkflowRun record SHALL be preserved.

#### Scenario: Delete draft
- **WHEN** user calls DELETE /drafts/{id}
- **THEN** system deletes the draft document and returns HTTP 204; the linked WorkflowRun is not deleted

### Requirement: Draft export as Markdown
The system SHALL allow exporting a draft as a downloadable Markdown file.

#### Scenario: Export draft as Markdown
- **WHEN** user clicks "导出 .md" or calls GET /drafts/{id}/export?format=markdown
- **THEN** system returns the draft content as a downloadable .md file with Content-Disposition: attachment header
